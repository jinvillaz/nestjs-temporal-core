    Function ChildWorkflow

    • ChildWorkflow(
          workflowType: Type<unknown>,
          options?: Record<string, unknown>,
      ): PropertyDecorator

      Property decorator to inject a child workflow proxy into a parent workflow. Child workflows are independent workflow executions that are started and managed by a parent workflow. This decorator creates a proxy that allows easy invocation of child workflows with proper typing.

      Parameters

      • workflowType: Type<unknown>

        The workflow class/type to create a proxy for

      • options: Record<string, unknown> = {}

        Optional configuration for the child workflow

        • taskQueue

          Task queue for the child workflow

        • workflowId

          Custom workflow ID pattern

        • parentClosePolicy

          Behavior when parent workflow closes

      Returns PropertyDecorator

      Example: Basic Child Workflow

      @Workflow()
      export class OrderWorkflow {
        @ChildWorkflow(PaymentWorkflow, { taskQueue: 'payments' })
        private paymentWorkflow: PaymentWorkflow;

        @ChildWorkflow(InventoryWorkflow)
        private inventoryWorkflow: InventoryWorkflow;

        @WorkflowRun()
        async execute(order: Order): Promise<OrderResult> {
          // Reserve inventory first
          const reservation = await this.inventoryWorkflow.reserve(order.items);

          // Process payment
          const payment = await this.paymentWorkflow.processPayment({
            amount: order.total,
            customerId: order.customerId
          });

          return { orderId: order.id, paymentId: payment.id };
        }
      }

      Example: Advanced Child Workflow Configuration

      @ChildWorkflow(NotificationWorkflow, {
        taskQueue: 'notifications',
        parentClosePolicy: 'ABANDON', // Continue even if parent fails
        workflowId: (parentId: string) => `notification-${parentId}`
      })
      private notificationWorkflow: NotificationWorkflow;

      Remarks

      • Child workflows are independent and can outlive the parent workflow
      • They provide better fault isolation and can be scheduled on different task queues
      • Use child workflows for operations that should continue even if parent fails
